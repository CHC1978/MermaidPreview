## 1. 安全漏洞 (Security Vulnerabilities)

### 1.1 [CRITICAL] openFile 路徑遍歷——缺乏根目錄邊界驗證

- **Location:** `src/MermaidPreview.cpp:811-857` (`OnOpenFileLink`)
- **Risk Level:** Critical
- **描述：**
  JS 端透過 `window.chrome.webview.postMessage({type:'openFile', path:'../../Windows/System32/evil.exe'})` 可觸發 `OnOpenFileLink`。C++ 端雖然使用 `GetFullPathNameW` 正規化路徑並呼叫 `GetFileAttributesW` 確認存在，但**未驗證最終路徑是否仍在當前文件的目錄樹內**。

  攻擊向量：惡意 `.md` 檔案中的連結如 `[click me](../../../Windows/System32/notepad.exe)` 可導致 EmEditor 開啟任意本機可執行檔（雖然不是執行，但在沙箱受限環境中可能利用 EmEditor 的 `Editor_LoadFileW` 的副作用）。

  另外，`filePath` 字串是從 WebMessage JSON 手動解析取得，JSON escape 解析未處理 `\uXXXX` 序列，惡意輸入可能繞過路徑驗證。

- **Exploit Scenario：**
  ```markdown
  [Secret](../../../sensitive/config.ini)
  ```
  使用者點擊後，EmEditor 開啟該檔案，洩露內容。

- **建議最小修復：**
  ```cpp
  // OnOpenFileLink 中，GetFullPathNameW 之後加入：
  // 確認 canonical[] 是 dir 的子路徑
  std::wstring canonicalStr = canonical;
  std::wstring dirCheck = dir + L"\\";
  // 正規化 dirCheck 也做一次 GetFullPathNameW
  WCHAR canonDir[MAX_PATH] = {};
  GetFullPathNameW(dirCheck.c_str(), MAX_PATH, canonDir, nullptr);
  std::wstring canonDirStr = canonDir;
  if (canonDirStr.empty() || canonicalStr.substr(0, canonDirStr.size()) != canonDirStr)
      return; // 路徑逃逸，拒絕
  ```

### 1.2 [CRITICAL] BunRenderer 回應 JSON 手動解析——SVG 注入至 HTML

- **Location:** `src/BunRenderer.cpp:347-440` (`RenderBlocks`)
- **Risk Level:** Critical
- **描述：**
  `RenderBlocks` 的 JSON 解析器是手動撰寫的字串搜尋邏輯，**不完整且可被欺騙**：

  1. **SVG 欄位逃脫序列不完整：** `\uXXXX` Unicode 逃脫未處理（`BunRenderer.cpp:387-395`），Bun 輸出的合法 JSON `<` (即 `<`) 會被原樣輸入 HTML，但更危險的是：若 SVG 字串中包含 `"` 前面有多個反斜線，如 `\\\"` (即 `\"`)，解析邏輯 `response[si] == '\\'` 會誤判邊界，導致取到超出實際 SVG 的字元。

  2. **無長度限制：** `svgStr` 沒有最大長度限制，Bun 惡意/崩潰輸出大量 SVG 時，`svgStr` 可無限成長導致 OOM。`response` 本身由 `ReadLine` 讀入，`m_readBuffer` 也沒有長度上限（見 HIGH-1.6）。

  3. **SVG 注入 HTML：** `r.svg` 被直接插入 HTML 字串（`MermaidPreview.cpp:715`），且 SVG 是 Bun 進程輸出——若 Bun 進程被污染（惡意 `mermaid.js` 替換），可任意注入 HTML/JS。雖然是本機攻擊面，仍需注意。

- **建議最小修復：**
  ```cpp
  // 在 svgStr 解析迴圈中加上長度限制
  constexpr size_t MAX_SVG_SIZE = 10 * 1024 * 1024; // 10MB
  if (svgStr.size() > MAX_SVG_SIZE) {
      r.error = L"SVG too large";
      break; // 放棄此 block
  }

  // 加入 \uXXXX 處理：
  else if (next == 'u' && si + 5 < response.size()) {
      // parse 4-hex-digit unicode
      std::string hexStr = response.substr(si + 2, 4);
      // ...convert and append
      si += 6;
  }
  ```

### 1.3 [HIGH] IsSafeUrl 未處理 null byte 與 URL-encoded 注入

- **Location:** `src/MarkdownParser.cpp:173-201` (`IsSafeUrl`)
- **Risk Level:** High
- **描述：**
  `IsSafeUrl` 只取前 24 個字元 lowercase 比對，但有幾個繞過向量：

  1. **null byte 注入：** `java\0script:` — `towlower` 不過濾 `\0`，而 24 字元擷取包含 null byte 會造成 `lower` 字串中間截斷（`std::wstring` 可含 null，但 `lower.substr(0, scheme.size())` 比對可能無效）。
  2. **URL-encoded scheme：** `%6A%61%76%61%73%63%72%69%70%74:` 不在黑名單中。雖然瀏覽器一般不執行 percent-encoded scheme，WebView2 可能有不同行為。
  3. **Unicode 全形字元：** `ｊａｖａｓｃｒｉｐｔ:` 等全形字元 `towlower` 無法正規化，可能繞過比對。
  4. **`blob:` scheme 未列入黑名單：** `blob:https://...` 可建立任意 Blob URL 執行 JS。

- **建議最小修復：**
  ```cpp
  // 1. 在取 lower 前，先掃描 null byte
  for (size_t j = i; j < url.size() && j < i + 24; j++) {
      if (url[j] == L'\0') return false; // null byte → 拒絕
      lower += (wchar_t)towlower(url[j]);
  }

  // 2. 加入 blob: 到黑名單
  static const std::wstring kDangerousSchemes[] = {
      L"javascript:", L"vbscript:", L"data:", L"blob:",
      L"file:", L"ms-appx:", L"ms-its:",
      L"mhtml:", L"ms-msdt:", L"ms-help:"
  };
  ```

### 1.4 [HIGH] WebMessage JSON 型別混淆——theme 訊息無 type 欄位驗證

- **Location:** `src/WebView2Manager.cpp:718-724`
- **Risk Level:** High
- **描述：**
  dispatch 邏輯以 `json.find(L"\"theme\"")` 匹配，而不是先解析 `"type"` 欄位：

  ```cpp
  // WebView2Manager.cpp:720
  if (json.find(L"\"theme\"") != std::wstring::npos && m_themeCallback) {
      bool dark = json.find(L"\"dark\":true") != std::wstring::npos;
      m_themeCallback(dark);
      return S_OK;  // ← 提前 return，後續 dispatch 都被跳過
  }
  ```

  任何包含 `"theme"` 字串的 JSON 都會觸發 theme callback 並提前返回。例如：
  ```json
  {"type":"edit","lineStart":0,"lineEnd":0,"newText":"hacked","theme":"x"}
  ```
  這條訊息會被誤判為 theme 訊息，`m_editCallback` 從不被呼叫。雖然不是直接安全漏洞，但破壞了訊息路由完整性。

- **建議最小修復：**
  在 dispatch 前先解析 `"type"` 欄位值：
  ```cpp
  // 提取 type 值
  std::wstring msgType;
  size_t typePos = json.find(L"\"type\":\"");
  if (typePos != std::wstring::npos) {
      typePos += 8;
      size_t typeEnd = json.find(L'"', typePos);
      if (typeEnd != std::wstring::npos)
          msgType = json.substr(typePos, typeEnd - typePos);
  }
  // 然後用 msgType == L"theme" 而非字串搜尋
  if (msgType == L"theme" && m_themeCallback) { ... }
  ```

### 1.5 [HIGH] ExtractMermaidJs 寫入前未驗證路徑合法性

- **Location:** `src/WebView2Manager.cpp:39-85` (`ExtractMermaidJs`)
- **Risk Level:** High
- **描述：**
  `m_resourceDir` 由 `SHGetFolderPathW(CSIDL_LOCAL_APPDATA)` 衍生，理論上安全，但：

  1. `CreateDirectoryW(parentDir, nullptr)` 和 `CreateDirectoryW(m_resourceDir, nullptr)` 的錯誤被完全忽略，若目錄創建失敗（權限問題、符號連結攻擊），`mermaidPath` 可能寫入錯誤位置。
  2. **Partial write 未處理：** `WriteFile` 後只檢查 `written == size`，但若 disk full 導致 partial write，關閉 handle 後檔案存在但內容不完整，下次啟動的 size check `> 100000` 可能仍通過（partial file 若 > 100KB），載入損壞的 JS。
  3. **Race condition（TOCTOU）：** `GetFileAttributesExW` 檢查後到 `CreateFileW(CREATE_ALWAYS)` 之間，其他進程可替換該路徑為符號連結，導致覆蓋任意文件（低可能性，但 LOCALAPPDATA 目錄通常用戶可寫）。

- **建議最小修復：**
  ```cpp
  DWORD written = 0;
  BOOL writeOk = WriteFile(hFile, data, size, &written, nullptr);
  CloseHandle(hFile);
  if (!writeOk || written != size) {
      // 刪除不完整檔案，避免殘留
      DeleteFileW(mermaidPath.c_str());
      return false;
  }
  ```

### 1.6 [HIGH] BunRenderer ReadLine 無緩衝區大小上限——OOM 風險

- **Location:** `src/BunRenderer.cpp:269-314` (`ReadLine`)
- **Risk Level:** High
- **描述：**
  `m_readBuffer` 持續 `append` 從 stdout pipe 讀取的資料，沒有任何大小限制。若 Bun 進程異常（崩潰後 pipe 仍開啟、無限輸出迴圈），`m_readBuffer` 可無限增長，最終 OOM 或嚴重記憶體壓力。

  此外，`ReadLine` 在 timeout 期間執行 `Sleep(10)` busy-polling，這會阻塞呼叫執行緒（雖然 `Start()` 有 5 秒 timeout，但 `RenderBlocks` 呼叫 `ReadLine` 的 timeout 可達 `5000 + blocks.size() * 3000` ms，高達數分鐘）。`RenderBlocks` 是在 UI 執行緒的 `UpdatePreview` 中同步呼叫，若 Bun 進程掛住，**整個 EmEditor UI 會凍結**。

- **建議最小修復：**
  ```cpp
  // ReadLine 加入緩衝區保護
  constexpr size_t MAX_BUFFER_SIZE = 20 * 1024 * 1024; // 20MB
  m_readBuffer.append(buf, bytesRead);
  if (m_readBuffer.size() > MAX_BUFFER_SIZE) {
      Stop(); // 異常進程，終止
      return "";
  }

  // RenderBlocks 應在背景執行緒呼叫（見 MEDIUM 建議）
  ```

### 1.7 [HIGH] _mmdInit 無 theme/look 參數驗證——JS 注入

- **Location:** `src/WebView2Manager.cpp:209-219` (`BuildHtmlPage` JS 內的 `_mmdInit`)
- **Risk Level:** High
- **描述：**
  `renderContent` 函式的 `theme` 參數直接從 WebMessage 傳入，最終透過 `_mmdInit` 注入到 `mermaid.initialize`。雖然 `RenderContent` 在 C++ 端只傳入 `darkMode ? L"dark" : L"light"`，但 `switchTheme` 函式中 `window.chrome.webview.postMessage({type:'theme',dark:dark})` 會觸發 C++ 重設 `m_bDarkMode`，然後 `m_bDarkMode ? L"dark" : L"light"` 是固定的，所以直接路徑上 theme 不可注入。

  **實際風險在於** `switchTheme` JS 函式直接呼叫 `_mmdInit(dark ? 'dark' : 'default')` 並重新渲染 mermaid，這裡的 `'dark'/'default'` 是 JS 內部固定字串，無法被外部注入。然而，`_mmdInit` 的 `look` 參數目前總是使用 `'classic'` hardcoded，若未來擴展 look 從 WebMessage 接收，需要在 C++ 端加入白名單驗證。

- **建議：** 在 C++ 端的 WebMessage handler 中，若未來加入 `look` 欄位處理，必須用白名單驗證（僅允許 `classic`, `neo`, `handDrawn`）再傳入 JS，不可直接 `EscapeForJS` 注入。

## 2. 資源洩漏分析 (Resource Leak Analysis)

### 2.1 [MEDIUM] BunRenderer Stop() 呼叫 TerminateProcess 後不等待足夠時間

- **Location:** `src/BunRenderer.cpp:233-253`
- **Root Cause：**
  `TerminateProcess` 後等待 3000ms，在高負載系統上可能不足。更重要的是，若 `m_hStdinWrite` 寫入時 Bun 進程已死亡（`WriteFile` 到斷裂 pipe），`WriteFile` 返回失敗但 handle 仍有效，`Stop()` 才關閉它——這是正確的。但若 `BunRenderer::~BunRenderer` 在 `Stop()` 之前（透過 shared_ptr 析構）被呼叫，且 `m_hProcess` 已是 nullptr（`Stop()` 被呼叫兩次），`TerminateProcess(nullptr, 0)` 會傳入 null handle——`TerminateProcess(nullptr)` 在 Windows 上是未定義行為（可能傳回錯誤，但偶爾崩潰）。

  ```cpp
  // BunRenderer.cpp:247
  if (m_hProcess) {
      TerminateProcess(m_hProcess, 0); // 正確，有 guard
  ```
  實際上有 null guard，此問題較低，但 double-Stop 在 EVENT_CLOSE_FRAME 流程中確實可能發生。

- **Fix Strategy：** 在 `Stop()` 開頭加入重入保護：
  ```cpp
  if (!m_bReady && !m_hProcess) return; // 已停止
  ```

### 2.2 [MEDIUM] BunRenderer EnsureSetup() 的子進程 handle 在 CreateProcessW 失敗時不洩漏，但超時後 hThread 已關閉，hProcess 未關閉的情況

- **Location:** `src/BunRenderer.cpp:127-143` (`EnsureSetup`)
- **Root Cause：**
  `WaitForSingleObject(pi.hProcess, 60000)` 後，`CloseHandle(pi.hProcess)` 和 `CloseHandle(pi.hThread)` 都有呼叫（BunRenderer.cpp:135-136），此路徑沒有洩漏問題。

  但若 `WaitForSingleObject` timeout（`bun install` 超過 60 秒），進程會繼續執行（未 `TerminateProcess`），handle 被關閉，但進程成為孤兒（orphan process）繼續執行並佔用資源。後續 `GetFileAttributesW(nodeModules)` 若仍失敗，`EnsureSetup` 返回 false，但孤兒 `bun install` 仍跑著。

- **Fix Strategy：**
  ```cpp
  DWORD waitResult = WaitForSingleObject(pi.hProcess, 60000);
  if (waitResult == WAIT_TIMEOUT) {
      TerminateProcess(pi.hProcess, 1); // 殺掉孤兒進程
  }
  CloseHandle(pi.hProcess);
  CloseHandle(pi.hThread);
  ```

### 2.3 [MEDIUM] WebView2Manager::Initialize 的非同步回調中 HRESULT 失敗無 early-out

- **Location:** `src/WebView2Manager.cpp:550-551`
- **Root Cause：**
  ```cpp
  controller->get_CoreWebView2(&m_webview);
  // 返回值被丟棄，若 get_CoreWebView2 失敗，m_webview 仍是 nullptr
  ```

## 3. 效能瓶頸 (Performance Bottlenecks)

### 3.1 [MEDIUM] RenderBlocks 在 UI 執行緒同步阻塞（最長數分鐘）

- **Location:** `src/MermaidPreview.cpp:651-664` (`UpdatePreview`)
- **Measured Impact：** 若 Bun 處理大圖（200+ 節點），`RenderBlocks` 的 timeout 為 `5000 + blocks * 3000` ms，EmEditor UI 完全凍結。
- **Root Cause：** `UpdatePreview` 從 `WM_TIMER` 回調（debounce timer）呼叫，在 Windows 訊息迴圈執行緒，`RenderBlocks` 同步等待 Bun 回應。
- **Optimization Plan：**
  - Short-term：將 timeout 降低至合理上限（如 10 秒），超時後 fallback 到 client-side render。
  - Long-term：將 `RenderBlocks` 移至背景執行緒（`std::async`），render 完成後透過 `PostMessage` 通知 UI 執行緒更新。

### 3.2 [MEDIUM] 大型 Markdown 文件的 ConvertToHtml 包含 O(n²) 字串連接

- **Location:** `src/MarkdownParser.cpp:538-962` (`ConvertToHtml`)
- **Root Cause：**
  ```cpp
  html += L"<li" + lineAttr + L">";  // 每次建立臨時 wstring
  html += ProcessInline(itemText);    // ProcessInline 遞迴呼叫，本身也是 O(n)
  ```
  `html` 雖有 `reserve(markdown.size() * 2)`，但 `ProcessInline` 對每個 `**bold**`、`*italic*` 等結構都呼叫 `text.substr(...)` 建立子字串。在 10k 行文件中，這導致大量記憶體配置，可能造成 200-500ms 解析時間。

- **Optimization Plan：**
  - Short-term：無 (已有 debounce 緩解)。
  - Long-term：改用索引範圍（`string_view` 或 `{begin, end}` 指標）代替 `substr`，避免複製。

### 3.3 [MEDIUM] renderedMermaidSrcs 快取：key 為 mermaid-placeholder-N（以 index 排列），切換 tab 後索引重設

- **Location:** `src/WebView2Manager.cpp:BuildHtmlPage` JS 內 (`renderContent` 函式，约 272-285 行)
- **Root Cause：**
  ```javascript
  var id = el.getAttribute('data-mermaid-id') || ('mmd-'+i+'-'+Date.now());
  if (renderedMermaidSrcs[id] && renderedMermaidSrcs[id].src === src) {
      el.innerHTML = renderedMermaidSrcs[id].svg; // 快取命中
  }
  ```
  快取 key 是 `mermaid-placeholder-0`、`mermaid-placeholder-1` 等固定字串，切換 tab 後重新渲染 HTML 時，若新文件的第 0 個 mermaid block 的 `src` 不同，快取不命中——這是正確行為。但若使用者在同一 tab 編輯，mermaid index 不變，只有 src 改變，快取也正確失效。**目前快取邏輯正確**，無問題。

### 3.4 [LOW] HasMermaidBlocks 在 TryAutoClose 中重複讀取文件內容

- **Location:** `src/MermaidPreview.cpp:609-616` (`TryAutoClose`)
- **Root Cause：**
  ```cpp
  if (!IsMarkdownFile(hwndView) || !HasMermaidBlocks(hwndView)) {
  ```

## 4. 程式碼品質 (Code Quality)

### 4.1 [MEDIUM] CMermaidFrame 成員 m_bSyncFromEditor / m_bSyncFromPreview 無原子性保護

- **Location:** `src/MermaidPreview.h:106-107`
- **描述：**
  `m_bSyncFromEditor` 和 `m_bSyncFromPreview` 在 Windows 訊息迴圈（UI 執行緒）設置，在 timer callback（同為 UI 執行緒）清除，理論上是單執行緒存取——無 race condition。但 `SetEditCallback` 的 lambda（`m_editCallback`）由 WebView2 的 `add_WebMessageReceived` 非同步回調觸發，在 COM apartment 執行緒中執行。若 WebView2 採用 STA（Single-Threaded Apartment），回調會被 marshal 回 UI 執行緒——因此**實際上安全**，但需確認 WebView2 的 apartment model。若未來移植或 WebView2 行為改變，需加入 `std::atomic<bool>` 保護。

- **建議：** 在開發文件中明確標注這些旗標只在 UI 執行緒存取，並加入 `static_assert` 或 comment 說明為何不需要 atomic。

### 4.2 [MEDIUM] BunRenderer.h 的 m_hProcess / m_hStdinWrite / m_hStdoutRead 使用 raw HANDLE

- **Location:** `src/BunRenderer.h`（未直接讀取，但從 .cpp 推斷）
- **描述：**
  三個 Windows HANDLE 使用 raw 型別，無 RAII 包裝。`Stop()` 必須手動關閉所有 handle，若新增異常路徑，容易洩漏。C++17 可使用 `std::unique_ptr<void, HandleDeleter>` 包裝：
  ```cpp
  struct HandleDeleter { void operator()(HANDLE h) { if(h && h!=INVALID_HANDLE_VALUE) CloseHandle(h); } };
  using UniqueHandle = std::unique_ptr<void, HandleDeleter>;
  UniqueHandle m_hProcess, m_hStdinWrite, m_hStdoutRead;
  ```

### 4.3 [MEDIUM] MarkdownParser::ProcessInline 巢狀遞迴可能 stack overflow

- **Location:** `src/MarkdownParser.cpp:425-429` (`ProcessInline`)
- **描述：**
  ```cpp
  out += ProcessInline(linkText); // 連結文字中的 inline 格式
  out += ProcessInline(text.substr(i + 3, end - i - 3)); // bold/italic 內容
  ```
  `ProcessInline` 遞迴呼叫自身，若 Markdown 包含深度嵌套格式（如 `**[***text***](url)**`），遞迴深度受限於 stack 大小。正常使用者文件不超過 5-6 層，但惡意構造的 `.md` 可能造成 stack overflow（`STATUS_STACK_OVERFLOW`，在 Windows 上會終止進程）。

- **建議最小修復：** 加入遞迴深度限制：
