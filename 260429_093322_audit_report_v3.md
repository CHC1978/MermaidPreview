## 修復引入的新問題分析

### NEW-001 [LOW] graveyard thread detach — 無生命週期上限
- **Location:** `MermaidPreview.cpp:491-494`
- **描述：** `CloseCustomBar` 中 `std::thread([f = std::move(m_renderFuture)]() mutable { f.wait(); }).detach()` 可防止 future destructor 阻塞 UI 執行緒，設計正確。但 detach 後，若使用者在 15s timeout 期間再次開啟 preview 並觸發 `RenderBlocks`，舊 graveyard thread 持有的 `shared_ptr<BunRenderer>` 仍活著，可能與新 `Start()` 呼叫並發存取 `m_hStdinWrite` / `m_hStdoutRead`（`ReadLine` 透過 handle 讀取）。
- **實際風險：** 低。`m_renderFuture` 中的 lambda 已捕獲 `renderer`（shared_ptr copy），`BunRenderer::Stop()` 在新 `Start()` 之前必定被呼叫（`~BunRenderer` destructor 或 `m_pBunRenderer.reset()`），故 pipe handles 已關閉，graveyard thread 的 `ReadLine` 會因 pipe broken 而短路返回 ""。但此假設依賴 `m_pBunRenderer.reset()` 先於 graveyard thread 的最後一次 `ReadFile` 完成，非強制 ordering。
- **建議：** 在 graveyard thread 的 lambda 加入 timeout：`if (f.wait_for(std::chrono::seconds(15)) != std::future_status::ready) { /* abandon */ }`。已有 15s RenderBlocks cap，故實際最壞情況已受限。

### NEW-002 [LOW] `msgType` 解析中 type 欄位若含 JSON escape 字元可能誤判
- **Location:** `WebView2Manager.cpp:743-748`
- **描述：** `msgType` 提取迴圈假設 type 值到下一個 `"` 為止（注意 comment 說明「JS senders only emit ASCII identifiers」）。若惡意頁面發送 `{"type":"the\\\"me","dark":true}`，迴圈在第一個 `"` 停止，`msgType = "the\\"` 不匹配任何 case，訊息被靜默丟棄。這是 fail-safe 行為（不觸發錯誤 callback），而非安全漏洞。但若未來 type 值有合法 escape，需要更完整的 JSON string parser。
- **風險：** Low — JS 端 type 值均為 ASCII 字面量（`"edit"`, `"scroll"`, `"theme"`），無 escape 字元。

### NEW-003 [MEDIUM] async lambda 捕獲 `renderer` 的 shared_ptr 可在 CMermaidFrame 析構後存活
- **Location:** `MermaidPreview.cpp:789-793`
- **描述：**
  ```cpp
  auto renderer = m_pBunRenderer;
  m_renderFuture = std::async(std::launch::async,
      [renderer, blocks = std::move(bunBlocks), theme]() {
          return renderer->RenderBlocks(blocks, theme);
      });
  ```
  `renderer` 是 `shared_ptr<BunRenderer>`，若 `CMermaidFrame` 析構（EmEditor 退出），`m_pBunRenderer` reset 後工作執行緒的 `renderer` copy 仍保持 `BunRenderer` 存活，`RenderBlocks` 繼續呼叫 `ReadLine`。這是設計意圖（graveyard pattern）。
  **問題：** `CloseCustomBar` 在 `m_renderFuture.valid()` 時 detach graveyard thread，但 `CMermaidFrame` 析構時不呼叫 `CloseCustomBar`（析構路徑直接 `m_renderFuture` destructor 阻塞）。若析構時 `m_renderFuture` 仍有效，`std::future::~future()` 會阻塞析構執行緒（通常是 DLL_PROCESS_DETACH）最多 15s。
- **建議：** 在 `CMermaidFrame` 析構函式（或 `OnDestroy` equivalent）中也執行 graveyard detach：
  ```cpp
  if (m_renderFuture.valid())
      std::thread([f = std::move(m_renderFuture)]() mutable { f.wait(); }).detach();
  ```
  確認 `DllMain(DLL_PROCESS_DETACH)` 路徑是否有此保護。
